The conflict resolution rule chose stage-order derivation over last-write-wins because device clocks drift and sync arrival order is arbitrary on a factory floor — neither is a trustworthy signal for production state. currentStage on a Bundle is never a value you overwrite directly; it is derived at write time from the highest stage recorded across all ScanLog entries, so no sync order can corrupt it. Backward movements — a scan claiming an earlier stage after a later one is already logged — are flagged with a reason and preserved rather than rejected, so no scan is ever discarded and a supervisor can review the anomaly without data loss.

At real scale I would move from SQLite to Postgres with a connection pool and migrate this into the NestJS module structure from the original architecture proposal with a proper service boundary. The localStorage deviceId would be replaced with server-registered device identities — hardware barcode scanners on a production floor need issued credentials, not browser-local UUIDs that reset when a tab is cleared. The sync engine would need retry backoff and partial-batch failure recovery so a single malformed scan does not block the rest of the queue. The flagged-bundle review flow would become an explicit supervisor screen rather than a visual badge.

The decisions that were mine: the stage-order conflict rule rather than last-write-wins, the clientId idempotency key on ScanLog as the deduplication primitive, the call to flag and preserve backward scans rather than reject them, and the scope cuts — no auth, no real device registration, web app instead of React Native as explicitly permitted by the brief. Codex assisted with scaffolding the API route handlers, the Dexie offline queue wiring, the seed script, and Tailwind layout. No backend business logic, schema design, or conflict resolution approach was AI-generated.
